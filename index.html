Git sem susto.

Repositório, commit, push, branch, merge... parecem sete palavras difíceis. Na real é só um fluxo. Clica em cada etapa e vê o que é — na ordem que acontece de verdade.

Antes do fluxo, o básico: o GitHub é tipo um Google Drive pra projetos — é onde o trabalho do time fica guardado na nuvem, com o histórico de tudo que mudou. O Git é a ferramenta que leva suas mudanças pra lá. O resto são só as etapas disso acontecer:

Repositório

A pasta do projeto, mas com memória.

É onde vive todo o trabalho do projeto. A diferença pra uma pasta comum: ela guarda cada versão e sabe quem mudou o quê. Dá pra voltar no tempo. Fica na nuvem (no GitHub) pra todo mundo do time acessar o mesmo lugar.

Pensa assim: um Google Drive que lembra de cada versão do arquivo e de quem mexeu — sem sobrescrever nada por acidente.
Branch

Uma cópia paralela pra você mexer à vontade.

Antes de mudar qualquer coisa, você cria um branch: uma versão separada onde dá pra testar sua ideia sem tocar no que está no ar. Se der ruim, ninguém foi afetado. Se der certo, você junta depois.

Pensa assim: um rascunho separado do documento oficial. Você rabisca à vontade; o original continua intacto até você decidir.
Commit

Salvar um marco do que você fez.

Cada commit é uma "foto" das suas mudanças, com um bilhete curto explicando o que mudou. Você faz vários ao longo do trabalho. Se algo quebrar, dá pra voltar pra qualquer foto anterior.

Pensa assim: aquele "salvar" do videogame com um checkpoint. Cada commit é um ponto seguro pra onde você pode voltar.

Na dúvida, é só pedir: "commita essas mudanças com uma mensagem que explique o que fiz"

Push & Pull

Mandar seu trabalho pro time (e receber o dos outros).

Enquanto você só commita, tudo fica na sua máquina. O push envia seus commits pro repositório na nuvem, pra todo mundo alcançar. O pull é o contrário: puxa o que os outros mandaram, pra você ficar em dia.

Pensa assim: push é subir seu arquivo pro Drive compartilhado; pull é baixar a versão mais nova que o colega subiu.
Pull Request (PR)

Pedir pra juntar seu trabalho — com revisão no meio.

Terminou seu branch? Você abre um Pull Request: um pedido formal de "quero juntar isso no oficial". O time olha, comenta, sugere ajuste e aprova. É aqui que entra a revisão — a mesma ideia da skill de revisão.

Pensa assim: mandar seu texto pro editor antes de publicar. Ninguém publica sozinho no que é importante — passa por outro olhar.
Merge

Juntar de vez. Sua mudança entra no ar.

Aprovado o PR, vem o merge: seu branch se funde no principal e a mudança passa a valer pra todo mundo. Fim da jornada — do rascunho ao oficial, com revisão no caminho.

Pensa assim: o editor aprovou e o texto foi publicado. O que era seu rascunho agora é a versão oficial.

Não precisa decorar nada disso. Na hora real, é só pedir: "me guia pra criar um branch, commitar e abrir um PR dessa mudança" — e o Claude te conduz etapa por etapa.

Anatomia de um bom pedido

As 5 partes que fazem funcionar

Contexto (quem você é) · Tarefa (o que quer, em verbo) · Formato (como receber) · Restrições (limites e conceitos, ex.: "não inventa nome") · Material (o insumo colado: transcrição, planilha, print).
